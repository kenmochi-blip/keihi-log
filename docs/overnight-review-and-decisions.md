## 3. B'案 実装設計（提案）

### 3.1 全体像
```
メンバーのブラウザ
   │ Googleサインイン（openid/email/profile のみ＝非機密・警告なし）で ID トークン取得
   │ Authorization: Bearer <Google ID token>
   ▼
Vercel Function（クリーンAPI / 窓口）  /api/data/*
   │ ① ID トークン検証（本人確認）
   │ ② alias→sheetId 解決（KV）
   │ ③ マスタ表でメンバー登録＆役割を確認（認可）
   │ ④ サービスアカウントでシート読み書き（google-auth-library）
   │ ⑤ 読み取りは KV に 30-60秒キャッシュ
   ▼
管理者のGoogleスプレッドシート（SAが編集者として共有されている）
```

### 3.2 クリーンAPI 仕様（案）
全て `Authorization: Bearer <ID token>` 必須。サーバーが本人＋役割を検証。

| メソッド/パス | 役割 | 内容 |
|---|---|---|
| `GET /api/data/expenses?month=YYYY-MM` | 全員（memberは自分の行のみ） | 経費一覧取得 |
| `POST /api/data/expenses` | admin/member | 経費申請（サーバー時刻で申請日時付与） |
| `PUT /api/data/expenses/:id` | 本人(申請済) / admin | 編集（修正履歴へ記録） |
| `DELETE /api/data/expenses/:id` | 本人(申請済) / admin | 削除（削除一覧へ移動） |
| `POST /api/data/expenses/:id/approve` | admin | 承認（J列） |
| `POST /api/data/expenses/:id/settle` | admin | 精算 |
| `GET /api/data/masters` | 全員 | メンバー/勘定科目/支払元/カスタムフラグ |
| `POST /api/data/masters` | admin | マスタ更新 |
| `GET /api/data/settings` | 全員 | 会社名・ヘッダー色・**Geminiキー**等 |
| `POST /api/data/receipt` | admin/member | 証票アップロード（SAが管理者フォルダに保存） |
| `GET /api/data/receipt/:id` | 権限者 | 証票取得（SA経由でストリーム） |

### 3.3 認証・認可
- **ID トークン検証**: Google公開鍵でJWT署名検証（`google-auth-library` の `OAuth2Client.verifyIdToken`）。`aud` が自分の CLIENT_ID であることを確認。→ なりすまし防止。
- **認可**: 検証済み email がマスタ表に存在するか＋役割（admin/member/viewer）をサーバーで判定し、エンドポイントごとに許可を強制。
- **重要**: 権限チェックの責任が Google共有 → 自前コードに移る。**ここはテストで担保必須**（§5）。

### 3.4 サービスアカウント
- GCPでSA作成 → JSON鍵を Vercel 環境変数 `GOOGLE_SA_KEY` に格納（コードに置かない）。
- 権限は対象シートの編集のみ（Driveの全体権限は付与しない）。
- **セットアップ時に自動共有**: 管理者のアプリが、作成したシート（drive.file対象）に Drive Permissions API で SA を editor 追加（`sendNotificationEmail=false`）。**新スコープ不要＝非機密のまま**。

### 3.5 KV スキーマ追加
- 既存: `alias:{code}→sheetId` 等。
- 追加候補: `alias_folder:{code}→folderId`（証票保存先）。
- キャッシュ: `cache:expenses:{sheetId}:{month}`（TTL 30-60s、書き込み時に無効化）。
- ※Geminiキー等の機密は KV に置かず、都度シートから読む（SA経由）方が安全。

### 3.6 必要な事前作業（あなたのGCP作業）
1. サービスアカウント作成＋JSON鍵発行。
2. Vercel に `GOOGLE_SA_KEY` 環境変数を設定。
3. （任意）SA用にSheets/Drive APIクォータの確認。

### 3.7 既存チームの移行
- B'導入前に作成済みのシートには SA が共有されていない。
- → 管理者が設定画面で「チーム同期を有効化」ボタンを1回押す（SAを共有する）導線が必要。

## 4. LP / FAQ の必要な変更（草案・未反映）

### 4.1 🔴 最重要：現在の「当社はデータを一切見ない」表現が B'と矛盾する
現在のLPは複数箇所で強く「当社のサーバーを通らない／当社には見えない」と謳っている：
- `index.html:1147-1148` 「データはすべて自分のGoogleアカウントへ（当社には見えない）」
- `index.html:1332-1334` 「当社がデータベースサーバーを持たない／直接保存」
- `index.html:1815-1816` 「データはすべてあなたのGoogleドライブへ（当社には保管しない）」

**B'（サービスアカウントプロキシ）では、データは当社Vercelを"通過"する**（保存はしないが、技術的にはSAが読める）。
→ **「一切見えない／通らない」は不正確になる。** 景表法・信頼の観点で要修正。

**推奨する書き換えの方向**:
- ❌「当社のサーバーを一切通りません」
- ⭕「データは**お客様のGoogleスプレッドシートに保存**され、当社のデータベースには**保存しません**。同期処理のため一時的に当社サーバーを経由しますが、保存・記録は行いません。」
- ⭕（上位オプション訴求）「データを当社サーバーに一切通さない**自社同期モード**（GAS方式）もご用意（こだわる企業向け）」

→ プライバシーポリシー（`privacy.html`）も同様に「処理のため一時的に経由する」旨の追記が必要。

### 4.2 権限・スコープ説明の更新（B'反映後）
- `faq.html Q202`（ログイン時の権限）: 現在 "email/Sheets/Drive" → B'では**メンバーは email/profile のみ**、管理者は +drive.file。要書き換え。
- `faq.html Q502`（証票の保存先）: B'では「SA経由で管理者のDriveフォルダ」に変わる可能性。要確認。
- `faq.html Q105/Q104`（役割）: B'では役割がGoogle共有でなく**アプリ側で厳格に制御**＝「閲覧者はシートを直接開けない」と明記できる（むしろ強みとして訴求可）。

### 4.3 訴求として強化できる点（B'のメリット）
